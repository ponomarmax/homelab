# Architecture

## Overview

Single-node local platform based on Docker Compose.

Host machine:
- Intel i7-10510U
- 8GB RAM
- 512GB SSD

The system combines infrastructure services, environmental data, and wearable sensor data into one repository and one evolving architecture.

## Wearable Architecture Direction

The wearable path is based on one collector app with multiple sensor adapters.

Initial:
- Polar Verity Sense adapter

Planned:
- Muse Athena adapter

The backend should not depend on one sensor-specific payload shape.
Instead, the system should rely on:
- stable transport metadata
- flexible sensor-specific payloads
- later normalization into analytical forms

## Separation of Concerns

### Collector App
Responsible for:
- connecting to sensors
- reading live data
- grouping data into upload units
- attaching shared metadata
- uploading batches
- showing debug state

### Ingestion API
Responsible for:
- accepting upload requests
- validating the outer envelope
- storing raw payloads durably
- acknowledging receipt
- exposing basic debug visibility

### Parsing / Normalization
Responsible for:
- sensor-specific parsing
- canonical analytical transformation
- preparing data for visualization and later feature extraction

### Analysis Layer
Responsible for:
- quality checks
- derived metrics
- epoching
- correlations
- future ML experiments

### Processing Layer

Responsible for:
- normalization
- window feature generation
- deterministic session summaries
- pipeline run metadata
- artifact traceability

The processing layer must remain deterministic and independently verifiable.

### Insights / Reporting Layer

Responsible for:
- prompt construction
- optional LLM interpretation
- LLM response validation
- report Markdown generation
- communication delivery such as Telegram

This layer is downstream of deterministic processing artifacts.

LLM calls and Telegram delivery must not be part of ingestion.
They also should not be required for normalization, feature generation, or session summary generation.

## Design Principles

- service isolation via Docker
- Docker Compose as the default deployment model
- minimal external exposure
- modular structure
- low resource footprint
- monitoring added early
- reusable repo structure for future services
- raw-first ingestion for wearable data
- ingestion is separate from parsing and analysis

## Storage

Persistent storage is required for:
- Home Assistant
- Prometheus
- Grafana
- raw wearable uploads
- normalized signal data
- future exported sessions

Docker volumes should be used by default for stateful services.
Large experimental datasets should not be committed to git.

### Practical implication

Dashboards are considered part of the system, not a visualization add-on.

This means:
- dashboards must work immediately after deploy
- dashboards must not depend on manual UI setup
- dashboards must be validated against real data
- dashboards must be safe to recreate together with Grafana

Future dashboard work should extend this structure rather than replace it.

## Wearable Ingestion Baseline V1

### Contract

- UploadChunkContract v1.0
- stream_type, source, collection are part of transport metadata
- X-User-ID header used for user separation
- fallback user_id = "1"

### Storage

- append-only JSONL
- structured path: user / source / date / session / stream_type
- one chunks.jsonl per session/stream

### Responsibilities

Ingestion API MUST:
- accept transport envelope
- persist raw data
- add server.received_at_server

Ingestion API MUST NOT:
- normalize timestamps
- compute features
- aggregate data
- interpret payload
